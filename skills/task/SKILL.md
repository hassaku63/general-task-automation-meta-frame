# task: タスク管理スキル

指定タスクに対して実装支援またはレビューを行う統合スキル。
タスクが未作成の場合は同梱テンプレートから4文書を自動生成する。

## テンプレート

- [PLAN テンプレート](templates/PLAN.md) - ゴール・課題の言語化
- [SPEC テンプレート](templates/SPEC.md) - 仕様の精緻化（自動化の境界を含む）
- [TODO テンプレート](templates/TODO.md) - タスク管理
- [KNOWLEDGE テンプレート](templates/KNOWLEDGE.md) - 知見の蓄積

## 使い方
```
/task {タスク名}
```

## 実行手順

### 1. タスクの準備
1. `$ARGUMENTS` からタスク名を取得する。引数がない場合はユーザーに聞く
2. タスク名をケバブケースに正規化する（例: `勤怠報告の自動化` → `attendance-report`）
3. `tasks/{task-name}/` の存在を確認する
   - **存在しない場合**: 上記テンプレート4文書を読み込み、`tasks/{task-name}/` に配置する。`{タスク名}` プレースホルダーをタスク名に置換する
   - **存在する場合**: そのまま次へ進む

### 2. モード選択
ユーザーに以下のモードを選んでもらう:
- **実装** — SPEC.md と TODO.md に基づき段階的に実装を進める
- **レビュー** — 4文書を品質チェックリストで評価する

---

## モード: 実装

### 実装 1. 文書の読み込みと確認
1. `tasks/{task-name}/` 配下の SPEC.md と TODO.md を読み込む
2. 文書が不十分な場合はレビューモードを先に実行することを提案する
3. SPEC.md の「自動化の境界」を確認し、AI が担当すべき部分を特定する

### 実装 2. フェーズごとの実装

TODO.md のフェーズに従い、1フェーズずつ進める。

各フェーズで:
1. フェーズ内のタスクをユーザーに提示する
2. 実装を進める
3. **判断ポイントに到達したら必ずユーザーに確認を求める**
4. フェーズ完了時に TODO.md のチェックボックスを更新する
5. 次のフェーズに進んでよいかユーザーに確認する

### 実装 3. 自動化の境界の厳守

SPEC.md の「自動化の境界」テーブルに従い:
- **AI 担当**: 自律的に実装を進める
- **人間担当**: ユーザーに作業を依頼し、完了を待つ
- **人間(判断)**: 選択肢を提示し、ユーザーの判断を仰ぐ

境界が曖昧な場合は、安全側に倒して人間に確認を求める。

### 実装 4. 成果物の配置
- スクリプトやコードは `tasks/{task-name}/` 内に作成する
- 再利用可能なものは `snippets/` への切り出しを提案する
- 調査結果は `docs/` への保存を提案する

### 実装 5. 実装完了時
1. TODO.md の全チェックボックスを更新する
2. SPEC.md の受け入れ基準を一つずつ確認する
3. KNOWLEDGE.md に以下を追記する:
   - 実装中に学んだこと
   - ハマりポイントと解決策
   - 再利用できそうなテクニック
   - 次回への改善案
4. レビューモードでの確認を案内する

---

## モード: レビュー

### レビュー 1. 文書の読み込み
1. `tasks/{task-name}/` 配下の4文書（PLAN.md, SPEC.md, TODO.md, KNOWLEDGE.md）をすべて読み込む
2. 文書が存在しない場合はその旨を報告する

### レビュー 2. チェックリストで評価

以下の観点で各文書を ○（十分）/ △（改善余地あり）/ ×（不十分）で評価する。

#### PLAN.md
- [ ] ゴールが具体的で測定可能か
- [ ] 現状の課題が明確か
- [ ] 理想の状態がゴールと整合しているか
- [ ] 制約・前提が漏れていないか

#### SPEC.md（最重視）
- [ ] インプット/アウトプットが明確か
- [ ] **自動化の境界が明確に定義されているか**
- [ ] AI 担当部分と人間担当部分が適切に分離されているか
- [ ] 判断ポイントの基準が具体的か
- [ ] 処理フローに抜け漏れがないか
- [ ] 受け入れ基準が検証可能か

#### TODO.md
- [ ] タスクが十分に分解されているか
- [ ] フェーズ分けが適切か
- [ ] 完了条件が明確か

#### KNOWLEDGE.md
- [ ] 実装後の場合: 学びが記録されているか
- [ ] 再利用可能な知見が特定されているか

### レビュー 3. レビュー結果の報告

以下の形式で報告する:

```
## レビュー結果: {タスク名}

| 文書 | 評価 | コメント |
|------|------|----------|
| PLAN.md | ○/△/× | ... |
| SPEC.md | ○/△/× | ... |
| TODO.md | ○/△/× | ... |
| KNOWLEDGE.md | ○/△/× | ... |

### 改善提案
1. ...

### 総合判断
実装に進めてよいか / 修正が必要か
```

### レビュー 4. 改善提案
- 評価が △ または × の項目について、具体的な改善案を提示する
- 特に SPEC.md の「自動化の境界」が曖昧な場合は、明確化を強く推奨する
